# VisionMark 静态官网部署文档

## 1. 当前部署结论

- 官网当前采用静态站部署，不在服务器上运行 Next.js 进程。
- 本地构建产物目录为 `out/`，服务器由 `nginx` 直接提供静态文件。
- 当前线上访问地址为 `https://www.visionmark.com.cn:8443`
- 当前方案不会改动服务器已有的 Node.js 18，也不会影响现有业务服务。

## 2. 当前服务器信息

- 服务器公网 IP：`81.70.220.123`
- 官网域名：`www.visionmark.com.cn`
- 静态文件目录：`/var/www/visionmark-landing`
- Nginx 配置文件：`/etc/nginx/conf.d/visionmark-landing.conf`
- HTTPS 证书目录：`/etc/letsencrypt/live/www.visionmark.com.cn/`

说明：

- 当前服务器 `SELinux` 状态为 `Disabled`
- 当前官网通过 `8443` 端口访问
- 当前 `8443` 端口由同一个 `nginx` 统一监听，再根据 `server_name` 分流

## 3. 为什么必须带 `:8443`

当前官网不是走默认的 `80` 或 `443` 端口，而是走：

- `https://www.visionmark.com.cn:8443`

因此下面这些地址当前都不是官网入口：

- `http://www.visionmark.com.cn/`
- `https://www.visionmark.com.cn/`

原因不是域名错误，而是服务器当前只对官网配置了 `8443` 监听。

## 4. 本地构建配置

项目已经切为静态导出模式，关键配置在：

- `next.config.ts`

当前关键配置为：

```ts
const nextConfig: NextConfig = {
  devIndicators: false,
  output: "export",
  images: {
    unoptimized: true,
  },
};
```

说明：

- `output: "export"`：构建后生成静态站产物到 `out/`
- `images.unoptimized: true`：避免依赖 Next.js 服务端图片优化

## 5. 首次部署步骤

### 5.1 本地构建

在项目根目录执行：

```powershell
npm run build
```

构建成功后会生成：

```text
out/
```

### 5.2 本地打包静态产物

```powershell
Remove-Item .\visionmark-landing-static.zip -ErrorAction Ignore
Compress-Archive -Path .\out\* -DestinationPath .\visionmark-landing-static.zip -Force
```

### 5.3 上传到服务器

```powershell
scp .\visionmark-landing-static.zip visionmark@81.70.220.123:/home/visionmark/
```

### 5.4 服务器解压部署

登录服务器后执行：

```bash
sudo mkdir -p /var/www/visionmark-landing
sudo rm -rf /var/www/visionmark-landing/*
sudo unzip -o /home/visionmark/visionmark-landing-static.zip -d /var/www/visionmark-landing
sudo chown -R nginx:nginx /var/www/visionmark-landing
sudo find /var/www/visionmark-landing -type d -exec chmod 755 {} \;
sudo find /var/www/visionmark-landing -type f -exec chmod 644 {} \;
```

说明：

- 不需要再处理 `chcon`
- 这台机器的 `SELinux` 已确认是 `Disabled`

## 6. 域名与证书

### 6.1 DNS

当前官网只绑定：

- `www.visionmark.com.cn`

DNS 需要有如下解析：

- 主机记录：`www`
- 记录类型：`A`
- 记录值：`81.70.220.123`

注意：

- DNS 只解析到 IP，不配置端口
- 端口 `8443` 是服务器 Nginx 侧的配置，不在 DNS 里填写

### 6.2 HTTPS 证书

当前证书使用 `certbot` + DNS TXT 验证方式申请。

申请命令：

```bash
sudo certbot certonly --manual --preferred-challenges dns -d www.visionmark.com.cn
```

证书生效路径：

- `/etc/letsencrypt/live/www.visionmark.com.cn/fullchain.pem`
- `/etc/letsencrypt/live/www.visionmark.com.cn/privkey.pem`

说明：

- `live/` 下的文件是固定路径
- 即使实际文件名是 `fullchain1.pem`、`privkey1.pem`，Nginx 也应始终引用 `live/.../fullchain.pem` 和 `live/.../privkey.pem`

## 7. Nginx 配置

当前官网配置文件：

```text
/etc/nginx/conf.d/visionmark-landing.conf
```

当前建议内容：

```nginx
server {
    listen 8443 ssl;
    http2 on;
    server_name www.visionmark.com.cn;

    root /var/www/visionmark-landing;
    index index.html;

    ssl_certificate /etc/letsencrypt/live/www.visionmark.com.cn/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/www.visionmark.com.cn/privkey.pem;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /_next/ {
        try_files $uri =404;
        access_log off;
        expires 30d;
        add_header Cache-Control "public, max-age=2592000, immutable";
    }

    location ~* \.(?:png|jpg|jpeg|gif|svg|ico|webp|woff2?|css|js)$ {
        try_files $uri =404;
        access_log off;
        expires 30d;
        add_header Cache-Control "public, max-age=2592000, immutable";
    }
}
```

检查与重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 10. 常见问题

### 10.1 为什么必须带 `:8443`

因为当前官网只监听：

- `8443`

而不是：

- `80`
- `443`

### 10.2 暴露 `8443` 会不会不安全

任何公网端口都会被扫描，`8443` 并不比 `443` 天然更危险。真正的安全重点是：

- 只暴露必要端口
- 公网前面只放 `nginx`
- 不把后端进程直接暴露到公网
- 定期更新系统和 Nginx

### 10.3 为什么不能直接在服务器上跑这个项目

当前服务器主业务仍依赖现有环境，直接升级全局 Node.js 版本有风险。  
因此当前官网采用静态导出方案，避免影响已有业务。

### 10.4 以后加功能会不会受影响

当前静态方案适合：

- 官网
- 落地页
- 纯展示页
- 前端交互型页面

如果未来要加这些能力，则建议重新切回 `Next + Node` 动态部署：

- Server Actions
- Next API Routes
- SSR
- 直接连数据库
- 服务端登录态
